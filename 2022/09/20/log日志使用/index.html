其他情况

一般我们的项目是由许多模块的，由不同人员开发的，那么不同的模块可能使用不同的日志，且又有可能使用不同的日志门面。

情况1

pro导入了A、B、C依赖，即pro项目中使用了A、B、C模块，其分别使用了不同的日志。

这种情况使用不会报错，每个模块的日志独立，使用的都是自己模块的日志框架，且使用的也是自己的配置文件。

但若pro配置了新的配置文件，如log4j.properties，则A会使用pro的，不会使用自己的。其他模块的也一样。

情况2

A、B、C分别使用不同日志，且都一样用了slf4j标准。注意logback本身就是实现了slf4j。

那么pro在调用三者时会报一个红色提示SLF4J: Class path contains multiple SLF4J bindings

因为pro导入了三个依赖，在三个slf4j实现中不知道选择哪个。

最终pro会自动选择一个（跟类的加载顺序有关），然后根据其配置文件输出。例如选择了log4j，那么输出的信息都是按照log4j的配置文件的。

解决：可以在pro导入A、B依赖时，排除掉slf4j的整合

情况3

A使用log4j，没有使用slf4j，B和C使用slf4j的log4j2和logback

此时pro调用，A的信息会按照自己配置的日志输出，而B和C的消息会从中选择一个slf4j，然后根据其配置文件输出。

那么怎么将A的log4j提升为slf4j标准呢

在pro导入A的依赖时，将其log4j依赖去除，然后添加一个依赖log4j-over-slf4j，升级成slf4j。

依次类推，我们也可以在B为log4j2，且没有实现slf4j时，升级成slf4j标准
