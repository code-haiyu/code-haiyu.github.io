docker

Docker 分为 CE 和 EE 两大版本。CE 即社区版（免费，支持周期 7 个月），EE 即企业版，强调安全，付费使用，支持周期 24 个月。

Docker CE 分为 stable test 和 nightly 三个更新频道。

官方网站上有各种环境下的 安装指南，这里主要介绍 Docker CE 在 CentOS上的安装。

docker容器

docker拉取的镜像是要使用的，所以docker为镜像提供运行容器，每个容器开启一个端口，然后接应宿主机的某个端口，通过访问服务器的某个端口，会被接应到对应的容器。

创建容器

一般docker官方都有创建容器的命令。

docker run 参数 镜像名

参数说明

  • -i：保持容器运行。通常与-t同时使用。加入it这两个参数后，容器创建后自动进入容器中，退出容器后，容器自动关闭
  • -t：为容器重新分配一个伪输入终端，通常与-i同时使用
  • -d：以守护（后台）模式运行容器。创建一个容器在后台运行，需要使用docker exec进入容器docker exec -it c2 /bin/bash。退出后，容器不会关闭
  • -it创建的容器一般称为交互式容器；-id创建的容器一般称为守护式容器
  • --name：为创建的容器命名

例：创建一个nginx容器

查看容器

docker ps [-a]

-a查看所有容器，包括停止的容器，否则查看的是允许的容器

查看日志

docker logs [-f] -f，follow，即控制台会持续跟踪日志

进入容器内部执行

docker exec -it containerName bash

-it：给当前进入的容器创建一个标准输入输出终端，允许我们与容器交互

containerName ：容器名

bash：进入后执行的目录，bash是一个linux终端交互命令

==可以通过这个命令进入容器内修改文件，例如nginx容器内的访问页，但是不推荐==

停止容器

docker stop 容器名

删除容器

docker rm [-f] 容器名 -f ，force 强制删除运行中的容器

查看容器信息

docker inspect 容器

数据卷

docker容器和数据耦合，如果我们要修改文件，就必须进入容器，修改非常不方便。

数据卷是一个虚拟目录，指向宿主机文件系统中的某个目录

数据卷基本语法

docker volume [command]

  • create 创建一个volume 例docker volume create xxx
  • inspect 显示一个或多个volume的信息
  • ls 列出所有的volume
  • prune 删除未使用的volume
  • rm 删除一个或多个指定的volume

数据卷挂载

创建的数据卷还没有挂载上容器的文件，可以通过运行容器时挂载

docker run --name 容器名 -p 宿主机端口:容器内服务端口 -v 数据卷:/容器内要挂载的文件夹路径 -d 镜像文件:版本tag

数据卷挂载，不过要修改文件就要通过查看数据卷信息来找到数据卷位置，进而修改

  • -v 数据卷:容器内要挂载的文件夹路径 ==若数据卷不存在时，则挂载时会帮你自动创建==

容器里的目录和文件也可以直接挂载宿主机目录

  • -v 宿主机目录:容器内要挂载的文件夹路径

  • -v 宿主机文件:容器内文件

例：创建html数据卷，将nginx容器的html文件夹挂载到上面

  • 创建html数据卷 docker volume create html

  • 运行容器：docker run --name myNginx -p 80:80 -v html:/usr/share/nginx/html -d nginx:latest

  • 查看数据卷信息docker volume inspect，里面有数据卷的位置

  • 查看数据卷的位置，进入后就可以看到文件，修改文件就可以同步到nginx的文件

数据卷继承

有三个容器c1、c2、c3，其中c3挂载到数据卷a1上，那么c1、c2可以继承c3的挂载规则，挂载到a1上。如果c3挂了，c1、c2的挂载也不会受影响。

docker run -it --volumes-from c3 --name c1 镜像

docker镜像原理

Linux文件系统由bootfs 和rootfs 两部分组成

  • bootfs：包含bootloader（引导加载程序）和kernel（内核）
  • rootfs：root文件系统，包含的就是典型的Linux 系统中的/dev、/proc、/bin等标准目录和文件
  • ==不同的Linux 发行版，bootfs 基本一样，而rootfs 不同，如ubuntu，CentOS等==

Docker 镜像是由特殊的文件系统叠加而成

最低端是bootfs，==并使用宿主机的bootfs==

第二层是root 文件系统rootfs ，称为base image

然后再往上可以叠加其他的镜像文件

一个镜像可以放在另一个镜像的上面。位于下面的镜像称为父镜像，最底部的镜像称为基础镜像

==镜像是只读的，当从一个镜像启动容器时，Docker会在最顶层加载一个读写文件系统作为容器，然后可以在镜像基础上添加新的东西，然后通过commit将容器转为新的镜像==

docker commit

commit可以在一个容器转为镜像

容器转镜像：docker commit 容器id 镜像名:版本号

镜像转成压缩文件：docker save -o 压缩文件名称 镜像名称:版本号

压缩文件加载成镜像：docker load -i 压缩文件名

例：在Ubuntu镜像上添加vim功能，然后转为新镜像

创建后可以docker images可以看到新的镜像。

但是制作新镜像一般不用commit，而是用dockerfile。

Dockerfile自定义镜像

我们上面的镜像都是别人做好的，如果我们需要自己制作镜像，则需要用Dockerfile来制作。dockerfile是一个文本文件，其中包含一个个指令，用来说明执行说明操作来构建镜像。

指令 说明
FROM 指定基础镜像
ENV 设置环境变量，可在看后面指令使用
COPY 复制本地文件到镜像的指定目录
ADD 添加文件到镜像中，可以来源于远程服务
RUN 执行Linux的shell命令，==在docker build时运行==
EXPOSE 指定容器运行时监听的端口，是给镜像使用者看的
ENTRYPOINT 镜像中应用的启动，容器运行（==docker run==）时调用
CMD 容器启动时执行的命令（==docker run时执行==）。当有多个CMD时，只有最后一个生效
MAINTAINER 标明这个dockerfile谁写的
LABLE 标签，最终可以在docker image中可以查看
WORKDIR 容器运行时内部的工作目录

CMD命令

CMD容器启动时执行的命令，==当有多个CMD时，只有最后一个生效==。

例如：从官网tomcat的Dockerfile看到，最后一行是**CMD [“catalina.sh”,”run”]**，即在docker run -p 8080:8080 tomcat:laster启动容器时，会执行这命令，从而启动tomcat。

但我们在启动容器时，指定其他命令，例docker run -p 8080:8080 tomcat:laster /bin/bash，则相当于在后面添加了**CMD[“/bin/bash”，”run”]**，导致前面的CMD失效，所以容器虽然启动了但也访问不到。

ENTRYPOINT命令

ENTRYPOINT和CMD一起用时，CMD的内容作为参数传递给ENTRYPOINT命令。例：通过Dockerfile构建nginx:test镜像

我们在docker run nginx:test 时，实际命令 nginx -c /etc/nginx/nginx.conf

当我们传参时 docker run nginx:test /etc/nginx/new.conf，实际命令 nginx -c /etc/nginx/new.conf

因为传的参数也是cmd命令，相当于在后面添加了**CMD[“/etc/nginx/new.conf”]**，会使前面的cmd命令失效，因为多个cmd时，最后一个生效。

例：基于ubuntu镜像制作镜像，运行一个docker-demo的java项目

  • 准备一个空目录，新建文件夹为Dockerfile
  • 将docker-demo.jar和jdk8.tar.gz拷贝到里面
  • 编写Dockerfile文件
  • 使用docker build -f dockerfile名 -t 镜像名 构建镜像
  • 使用docker run创建容器并运行

==上面Dockerfile中定义的各种路径，是指在容器内部的路径，容器内部也是一个小linux系统，有一样的文件夹系统==

上面Dockerfile文件里的jdk安装已经有人帮我们做好了，我们直接引入镜像就行

例：自定义centos7镜像，默认登录路径为/usr，可以使用vim

虚悬镜像

镜像没有仓库名或没有标签，在使用docker image查看镜像时，有的镜像的仓库和标签为

例如我们push的java：1.8镜像，原本是有镜像名和标签的，随着官方镜像维护，发布了新版本后，重新 docker pull java:1.8 时， java:1.8 这个镜像名被转移到了新下载的镜像身上，而旧的镜像上的这个名称则被取消，从而成为了 。除了 docker pull 可能导致这种情况，docker build 也同样可以导致这种现象。由于新旧镜像同名，旧镜像名称被取消，从而出现仓库名、标签均为的镜像。

查看虚悬镜像docker images -f dangling=true

删除虚悬镜像docker rmi $(docker images -q -f dangling=true)

docker-network

作用

容器间的互联和通信以及端口映射

容器ip变动时可以通过服务名来网络通信

常用命令

docker network ls 查看网络，当我们安装docker后，默认会自动创建三个网络bridge，host，none

dcoker network --help 查看网络命令帮助

docker-network网络模式

网络模式 介绍
bridge 默认为该模式，为每一个容器分配、设置ip，并将容器连接到一个docke0网桥，保证主机与容器是一个物理网络（类似VMware的NAT）
host 容器使用宿主机的ip和端口
none 容器有独立的network namespace，但并没有进行任何网络设置
container 新创建的容器会和一个指定容器共享ip，端口

在运行镜像成容器后，可以通过docker inspect 容器名来查看容器ip和网络模式，也可以docker exec -it 容器 bash 进入容器，然后ip addr查看ip

自定义网络

容器实例内默认网络ip生产规则，例如：启动u1，u2两个容器，ip分别为192.168.0.1和192.168.0.2，若u2宕机了，这时u3启动，则u3的ip会是192.168.0.2，这样我们访问时会是u3的服务。另外我们在容器里ping另一个容器的ip，是ping得通的，但是ping容器名，例如ping n2，是ping不通的。

所以我们需要自定义网络，通过服务名来互相访问，这样即使ip地址改了，也可以访问得到正确的容器。

步骤

创建网络 docker network create 网络名 默认是bridge模式

查看docker network ls

启动容器时加上参数docker run ... --network 网络名

这样在同一个网络里的容器是ping 容器名是能成功的。
